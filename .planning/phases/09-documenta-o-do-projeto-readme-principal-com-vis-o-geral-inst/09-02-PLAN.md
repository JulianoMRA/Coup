### docs/CONTRIBUTING.md (per D-12, D-13, D-14):

1. **Titulo:** `# Como Contribuir`

2. **Setup local (per D-12):**
   Referenciar INSTALL.md com link: "Para configurar o ambiente de desenvolvimento, siga as instrucoes em [Instalacao e Execucao Local](INSTALL.md)."
   NAO duplicar as instrucoes de instalacao.

3. **Convencoes de codigo:**
   - TypeScript em todo o projeto
   - Testes sao obrigatorios para novas funcionalidades

4. **Padrao de commits (per D-13):**
   Conventional Commits. Exemplos:
   ```
   feat: adiciona timer na janela de reacao
   fix: corrige desconexao durante troca de cartas
   chore: atualiza dependencias
   refactor: extrai logica de validacao para helper
   test: adiciona testes para o fluxo de bloqueio
   ```

5. **Hooks automaticos (per D-13):**
   - O projeto usa **husky** com **secretlint** no pre-commit
   - O hook roda automaticamente a cada commit — verifica se nenhum secret foi commitado acidentalmente
   - **NUNCA use `git commit --no-verify`** — isso desabilita a unica protecao automatica contra vazamento de secrets

6. **Antes de abrir um PR (per D-13):**
   ```bash
   npm run typecheck
   npm run test
   ```
   Ambos devem passar sem erros. O CI roda os mesmos checks — se falharem localmente, vao falhar no CI tambem.

7. **Fluxo de Pull Request (per D-14):**
   1. Crie uma branch com nome descritivo (ex: `feat/timer-reacao`, `fix/desconexao-troca`)
   2. Faca commits atomicos seguindo Conventional Commits
   3. Abra um PR com descricao clara do que muda e por que
   4. Aguarde todos os checks (CI) passarem
   5. Solicite review e aguarde aprovacao antes do merge

### docs/DEPLOY.md (per D-15, D-16, D-17, D-18, D-19):

1. **Titulo:** `# Deploy em Producao`

2. **Visao geral (per D-15):**
   O deploy e feito em duas etapas: **backend primeiro (Railway), depois frontend (Vercel)**. A ordem importa porque o frontend precisa da URL do backend.

3. **Variaveis de ambiente (per D-19):**
   Referenciar `.env.example`: "Consulte o arquivo [`.env.example`](../.env.example) para a descricao completa de cada variavel."
   NAO duplicar o conteudo.

4. **Etapa 1: Backend no Railway (per D-16):**
   Documentar passo a passo:
   1. Crie um projeto no [Railway](https://railway.com) e conecte ao repositorio GitHub
   2. O campo **Root Directory** deve ficar **em branco** (padrao) — o monorepo precisa da raiz para resolver as dependencias entre pacotes (packages/shared)
   3. O `railway.json` na raiz do repositorio configura automaticamente:
      - Build: `npm run build` (compila shared e backend)
      - Start: `node dist/index.js`
   4. Configure a variavel de ambiente `FRONTEND_URL` com a URL da Vercel (pode ser atualizada depois)
   5. **NAO configure `PORT`** — o Railway injeta automaticamente a porta correta
   6. Apos o deploy, copie a URL publica gerada (formato `https://xxx.up.railway.app`)

5. **Etapa 2: Frontend na Vercel (per D-17):**
   1. Crie um projeto na [Vercel](https://vercel.com) e conecte ao repositorio GitHub
   2. O `vercel.json` na raiz configura automaticamente o build do Next.js
   3. Configure a variavel de ambiente `NEXT_PUBLIC_BACKEND_URL` com a URL do Railway obtida na Etapa 1
   4. O deploy e automatico a cada push para `main`
   5. Apos o deploy, copie a URL publica da Vercel
   6. Volte ao Railway e atualize `FRONTEND_URL` com a URL da Vercel (se ainda nao configurou)

6. **Alerta importante (per D-18):**
   Incluir um bloco de destaque (blockquote com emoji ou negrito):
   > **Atencao:** Variaveis `NEXT_PUBLIC_*` sao embeddadas no bundle durante o `next build`. Se voce alterar `NEXT_PUBLIC_BACKEND_URL` na Vercel, e necessario disparar um novo deploy (Redeploy) para o novo valor entrar em vigor.
