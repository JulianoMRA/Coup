1. **Titulo:** `# Arquitetura do Sistema`

2. **Diagrama Mermaid (per D-09, D-10):**
   Incluir o seguinte diagrama em um bloco ```mermaid:

   ```mermaid
   graph LR
       Browser["Navegador\n(React/Next.js)"]
       Frontend["apps/frontend\nNext.js :3000"]
       Backend["apps/backend\nExpress + Socket.IO :3001"]
       GameState["GameState\n(Map em memoria)"]
       Shared["packages/shared\nTypeScript types"]

       Browser -->|"HTTP (paginas)"| Frontend
       Browser <-->|"WebSocket (Socket.IO)"| Backend
       Frontend <-->|"Socket.IO client"| Backend
       Backend --> GameState
       Shared -.->|"tipos compartilhados"| Frontend
       Shared -.->|"tipos compartilhados"| Backend
   ```

3. **Visao Geral (per D-11):**
   Paragrafo explicando que o Coup Online usa uma arquitetura monorepo com dois servicos e uma biblioteca de tipos compartilhados. A comunicacao em tempo real entre os jogadores e feita via WebSocket (Socket.IO). O estado do jogo e mantido 100% em memoria no servidor (sem banco de dados no v1).

4. **Estrutura do Monorepo (per D-11):**
   Arvore de diretorios mostrando:
   ```
   /
   +-- apps/
   |   +-- frontend/       -> Next.js, React, Tailwind CSS, Socket.IO client
   |   +-- backend/        -> Express, Socket.IO, TypeScript
   +-- packages/
   |   +-- shared/         -> tipos TypeScript compartilhados
   +-- .env.example        -> template de variaveis de ambiente
   +-- railway.json        -> config de deploy do backend (Railway)
   +-- vercel.json         -> config de deploy do frontend (Vercel)
   +-- package.json        -> workspace root, scripts globais
   ```

5. **Camadas da Aplicacao (per D-11):**
   Descricao textual de cada camada:
   - **Frontend (apps/frontend):** Interface do jogo em React/Next.js. Se conecta ao backend via Socket.IO para receber atualizacoes de estado em tempo real. Serve paginas estaticas via HTTP.
   - **Backend (apps/backend):** Servidor Express com Socket.IO. Gerencia salas de jogo, processa acoes dos jogadores atraves de uma FSM (maquina de estados finita), e broadcast de estado para todos os jogadores conectados.
   - **Shared (packages/shared):** Tipos TypeScript usados por ambos os servicos — definicoes de cartas, estados do jogo, eventos Socket.IO e tipos do lobby. Garante tipagem consistente entre frontend e backend.
   - **Estado em Memoria:** O estado de cada partida e armazenado em um `Map<roomId, GameState>` na memoria do servidor. Nao ha banco de dados — o estado e perdido se o servidor reiniciar. Decisao intencional para simplicidade no v1.

6. **Comunicacao (per D-11):**
   - Paginas e assets: HTTP convencional (Next.js serve o frontend)
   - Estado do jogo em tempo real: WebSocket bidirecional via Socket.IO
   - O frontend envia acoes do jogador (ex: jogar Income, desafiar) e recebe o estado atualizado projetado para aquele jogador especifico (cartas ocultas dos outros jogadores nao sao enviadas)
