## Summary

This phase is a pure frontend visual redesign: no new logic, no socket events, no new npm packages. All 12 component files plus 2 page files need to be restyled in place. The architecture is already sound — the goal is to apply a dark card-game aesthetic on top of the existing structure.

The stack is Tailwind v4 (CSS-first, `@theme` directives in `globals.css`) with shadcn/ui components (zinc/dark base). Custom character colors are added as new `@theme` CSS variables in `globals.css` — no `tailwind.config.ts` exists or is needed. The `cn()` utility (clsx + tailwind-merge) is already in use throughout all components and is the correct pattern for conditional class application.

Two new components must be created: `CharacterCard` (reusable card face/back, used in "Minhas Cartas" and InfluenceCardSelector) and `PlayerSeat` (replaces the flat list rows in PlayerPanel). All other files are restyled in-place without changing props or socket logic.

**Primary recommendation:** Create `character-card.tsx` and `player-seat.tsx` as new files; restyle all existing components using Tailwind utility classes and the `cn()` pattern; extend `globals.css` with 5 character color tokens under `@theme`.

### Pattern 1: Character Color Tokens via `@theme` in globals.css

Tailwind v4 uses CSS-first configuration. Custom colors are declared in `globals.css` under the `@theme` block and become available as Tailwind utility classes automatically. [VERIFIED: STATE.md — "Tailwind v4 is CSS-first — no tailwind.config.ts"]

```css
/* Source: globals.css @theme extension */
@theme {
  /* Existing tokens stay unchanged */

  /* Character color tokens — added for Phase 06.1 */
  --color-character-duke: hsl(271 76% 53%);        /* purple-600 range */
  --color-character-captain: hsl(217 91% 60%);     /* blue-500 range */
  --color-character-assassin: hsl(0 84% 60%);      /* red-500 range */
  --color-character-ambassador: hsl(142 71% 45%);  /* green-500 range */
  --color-character-countess: hsl(330 80% 65%);    /* pink-500 range */
}
```

These tokens become Tailwind classes: `bg-character-duke`, `text-character-duke`, `border-character-duke`, etc.

**Key insight on "revealed" semantics:** In the existing codebase, `card.revealed = true` means the card has been flipped face-up (eliminated/challenged). When `revealed = false`, the card is in-hand (face-down to opponents, face-up to owner). The `CharacterCard` component needs two usage modes:

### Pattern 6: Tailwind v4 Arbitrary Values

Since Tailwind v4 has no config file, one-off values use square bracket syntax. This is available for all utility classes:

```
bg-[hsl(271_76%_53%/0.15)]   ← character color tint background
ring-[hsl(271_76%_53%)]       ← character color ring
```

However, the `@theme` token approach (Pattern 1) is preferable for the 5 character colors since they're used repeatedly.

### Anti-Patterns to Avoid

- **Do not modify `src/components/ui/`** — shadcn UI primitives are regenerated by the shadcn CLI and manual edits will be lost.
- **Do not add `tailwind.config.ts`** — Tailwind v4 is CSS-first. Adding a config file will conflict. [VERIFIED: STATE.md]
- **Do not use emojis** — locked decision, use Lucide icons only.
- **Do not use `Dagger` icon** — MISSING from lucide-react 0.511.0. Use `Sword` for Assassin. [VERIFIED: npm registry probe]
- **Do not change props interfaces** — ActionBar, PlayerPanel etc. props stay the same; only markup/classes change.
- **Do not remove `"use client"` directives** — all components use hooks or event handlers that require client-side rendering.
- **Do not use inline `style={{}}` for character colors** — use Tailwind classes via `@theme` tokens for consistency with the design system.

## Common Pitfalls

### Pitfall 1: Tailwind v4 Token Naming Conflicts
**What goes wrong:** Adding a token named `--color-character-duke` but trying to use it as `bg-duke` (wrong prefix).
**Why it happens:** Tailwind v4 strips `--color-` prefix to create utility class names. So `--color-character-duke` → `bg-character-duke`. Using the wrong class name silently produces no styling.
**How to avoid:** Use the full prefix: `bg-character-duke`, `text-character-duke`, `border-character-duke`.
**Warning signs:** No color applied in browser, no error in build.

### Pitfall 2: `revealed` Field Semantic Confusion
**What goes wrong:** Treating `card.revealed = false` as "hidden from everyone" when in context of "Minhas Cartas", `revealed = false` means the card is active (owner should see face).
**Why it happens:** The field means "has this card been publicly revealed (eliminated/challenged)?", not "is this card visible to the current user."
**How to avoid:** In `CharacterCard`, the `revealed` prop controls greyed-out state (true = eliminated look), not whether the face is shown. Pass face content always for the player's own hand; pass card back only when rendering *other* players' hidden cards (which never happens — `ClientGameState` only contains `cardCount` for opponents).
**Warning signs:** Player can't see their own cards; or eliminated cards look the same as active cards.

### Pitfall 3: Fixed Footer Z-Index Conflicts
**What goes wrong:** `ConnectionBadge` (fixed bottom-right, z-50) overlaps the action bar buttons on mobile.
**Why it happens:** Both use `fixed` positioning; ConnectionBadge is currently `bottom-4 right-4` which lands on top of the action bar.
**How to avoid:** Move ConnectionBadge to the header during game phase (it already sits in the GameBoard header). Review z-index layering for all fixed elements: WinnerOverlay (z-50), ConnectionBadge (z-50), action bars (no z-index specified).
**Warning signs:** Buttons in bottom-right of action bar unclickable on small screens.

### Pitfall 4: Hardcoded `max-h-[400px]` on GameLog
**What goes wrong:** GameLog has `max-h-[400px]` hardcoded; on larger screens this underutilizes space; on smaller screens it may still overflow.
**Why it happens:** Current implementation doesn't use flex height properly inside the Card.
**How to avoid:** For desktop, the GameLog should fill available height in the flex layout. Use `flex-1 overflow-y-auto` with the parent having a defined height.
**Warning signs:** Log area looks cramped while the left panel has empty space.

### Pitfall 5: Mobile Action Bar Overflow
**What goes wrong:** ActionBar has two rows of 4 buttons each — on mobile (375px), this overflows horizontally.
**Why it happens:** Buttons are in `flex` row with `gap-4`; 8 buttons don't fit two rows of 4 on narrow screens.
**How to avoid:** On mobile, use a grid layout (`grid grid-cols-2 gap-2`) or reduce button text. Min touch target is 44px per CONTEXT.md. The character action buttons (row 2) can use icon + short label.
**Warning signs:** Horizontal scroll on action bar on iPhone.

### Pitfall 6: Adding `tailwind.config.ts`
**What goes wrong:** Creating a Tailwind config file to add custom colors invalidates the CSS-first setup.
**Why it happens:** Developers familiar with Tailwind v3 instinctively reach for the config file.
**How to avoid:** All customization goes in `globals.css` under `@theme`. [VERIFIED: STATE.md — locked decision]
**Warning signs:** Build error or conflicting config warning.
